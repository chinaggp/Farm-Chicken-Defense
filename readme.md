# 《守护小鸡》小游戏完整技术方案（一期）

> 项目代号：Farm Chicken Defense  
> 项目类型：休闲 / 画线防御 / 物理解谜  
> 发布平台：微信小游戏、抖音小游戏  
> 技术方向：无服务器、低成本、快速上线验证收益  
> 一期目标：30 个关卡 + 基础广告变现 + 双平台可提审版本

---

# 1. 项目概述

## 1.1 项目背景

项目目标是开发一款类似《Save The Pets》的轻度休闲小游戏，但采用完全不同的主题包装与玩法细节，降低版权风险，并形成自身特色。

核心玩法：

- 小鸡会受到惊吓乱跑
- 老鹰会高速俯冲攻击
- 玩家通过画木栅栏或藤蔓进行保护
- 坚持到倒计时结束即可通关

## 1.2 一句话描述

玩家需要快速搭建木栅栏或藤蔓，保护受到惊吓而乱跑的小鸡，阻挡老鹰俯冲攻击，并坚持到倒计时结束。

## 1.3 一期边界

一期只做可上线验证版本，不做重度长期系统。

一期包含：

- 30 个固定关卡
- 画线生成防御体
- 小鸡随机移动
- 老鹰巡逻、锁定、俯冲攻击
- 胜负判定
- 星级评价
- 本地关卡进度
- 激励视频广告
- 插屏广告
- 微信小游戏与抖音小游戏构建适配

一期不包含：

- 自建后端服务
- 自建数据库
- 联网排行榜
- 账号系统
- 复杂皮肤商城
- 云存档
- 多人玩法
- 关卡编辑器

# 2. 项目目标

## 2.1 产品目标

- 完成 30 个关卡
- 单局时长控制在 20 到 60 秒
- 新手前 3 关无需说明即可理解玩法
- 首次进入 10 秒内可开始第一局
- 广告点位不阻断核心体验

## 2.2 商业目标

验证：

- 用户点击率
- 次日留存
- 激励广告收益
- 插屏广告收益
- 用户平均在线时长
- 关卡失败后的广告复活转化率

## 2.3 技术目标

- 不购买服务器
- 不自建数据库
- 关卡数据本地配置化
- 核心玩法逻辑可维护、可调参
- 平台能力通过适配层隔离
- 同一套主工程可分别构建微信小游戏与抖音小游戏

# 3. 技术选型

## 3.1 客户端引擎

使用 Cocos Creator 3.8.8。

原因：

- 对微信小游戏、抖音小游戏支持成熟
- 适合 2D 物理小游戏
- TypeScript 工程可维护性较好
- 构建体积和性能可控

## 3.2 开发语言

- TypeScript：游戏业务逻辑
- JSON：关卡配置、数值配置、广告配置
- Cocos Prefab：角色、障碍物、UI 组件

## 3.3 物理方案

一期推荐使用 Cocos 内置 2D 物理能力。

使用范围：

- 防御线碰撞体
- 小鸡碰撞检测
- 老鹰俯冲碰撞检测
- 场景边界碰撞

不建议一期自行实现完整物理引擎。画线防御玩法的难点在于手感、碰撞稳定性和关卡调参，优先使用引擎能力可以降低实现风险。

## 3.4 数据存储

一期只使用平台本地存储：

- 微信：`wx.setStorageSync` / `wx.getStorageSync`
- 抖音：`tt.setStorageSync` / `tt.getStorageSync`
- 编辑器或 Web 调试：`localStorage`

本地存储内容：

- 当前解锁关卡
- 每关最高星级
- 音效开关
- 新手引导完成状态
- 广告冷却时间

# 4. 游戏核心玩法

## 4.1 核心循环

```text
进入关卡
↓
小鸡开始乱跑
↓
老鹰开始盘旋
↓
玩家画木栅栏
↓
限制小鸡移动
↓
阻挡老鹰俯冲
↓
倒计时结束
↓
通关
```

## 4.2 单局流程

```text
加载关卡配置
↓
生成地图、边界、小鸡、老鹰、障碍物
↓
倒计时准备
↓
玩家开始画线
↓
画线结束后生成物理防御体
↓
小鸡执行随机移动
↓
老鹰按阶段巡逻与俯冲
↓
检测碰撞与胜负
↓
展示结算界面
↓
进入下一关或重试
```

## 4.3 胜负规则

胜利条件：

- 小鸡在倒计时结束时仍存活

失败条件：

- 老鹰命中小鸡
- 小鸡掉出安全区域
- 特定关卡中小鸡接触危险区域

复活规则：

- 失败后允许观看一次激励视频复活
- 复活后回到失败前的关卡初始状态，不保留旧防御线
- 每局最多复活一次，避免广告滥用影响体验

## 4.4 星级评价

建议一期采用简单规则：

- 3 星：一次通过，且剩余画线长度大于 30%
- 2 星：一次通过，但剩余画线长度小于等于 30%
- 1 星：复活后通过

星级只用于关卡列表展示，不作为强制卡点。

# 5. 系统架构设计

## 5.1 分层结构

```text
GameApp
├─ Core
│  ├─ GameManager
│  ├─ LevelManager
│  ├─ StateMachine
│  └─ EventBus
├─ Gameplay
│  ├─ ChickenController
│  ├─ EagleController
│  ├─ DrawLineController
│  ├─ DefenseLineFactory
│  └─ CollisionHandler
├─ Config
│  ├─ LevelConfigLoader
│  ├─ GameBalanceConfig
│  └─ AdConfig
├─ UI
│  ├─ HomeView
│  ├─ LevelSelectView
│  ├─ GameHudView
│  ├─ ResultView
│  └─ SettingsView
├─ Platform
│  ├─ PlatformAdapter
│  ├─ WeChatAdapter
│  ├─ DouyinAdapter
│  └─ WebAdapter
└─ Services
   ├─ StorageService
   ├─ AdService
   ├─ AudioService
   └─ AnalyticsService
```

## 5.2 核心原则

- 游戏玩法逻辑不直接调用 `wx` 或 `tt`
- 广告、存储、分享、震动等平台能力统一走 `PlatformAdapter`
- 关卡内容尽量配置化，减少硬编码
- UI 只展示状态，不直接决定胜负逻辑
- 物理碰撞统一汇总到 `CollisionHandler`，避免散落在多个脚本中

# 6. 核心模块设计

## 6.1 GameManager

职责：

- 控制游戏主状态
- 管理单局开始、暂停、恢复、胜利、失败、结算
- 协调 UI、关卡、广告、音效

建议状态：

```ts
enum GameState {
  Loading = 'Loading',
  Ready = 'Ready',
  Playing = 'Playing',
  Paused = 'Paused',
  Win = 'Win',
  Lose = 'Lose',
  Revive = 'Revive',
  Result = 'Result'
}
```

## 6.2 LevelManager

职责：

- 加载关卡配置
- 生成关卡实体
- 清理关卡对象
- 提供当前关卡运行参数

关卡配置建议使用 JSON：

```json
{
  "id": 1,
  "name": "First Fence",
  "duration": 25,
  "maxDrawLength": 600,
  "chickens": [
    {
      "x": 0,
      "y": -120,
      "moveSpeed": 80,
      "panicLevel": 1
    }
  ],
  "eagles": [
    {
      "x": 0,
      "y": 260,
      "patrolRadius": 180,
      "attackInterval": 4,
      "attackSpeed": 500
    }
  ],
  "staticObstacles": [],
  "dangerZones": []
}
```

## 6.3 DrawLineController

职责：

- 监听触摸开始、移动、结束
- 采样触摸点
- 限制最大画线长度
- 过滤过密点位
- 生成线条显示
- 在触摸结束后创建物理碰撞体

关键规则：

- 最小采样距离建议 8 到 12 像素
- 最大画线长度由关卡配置控制
- 画线不能穿过禁止绘制区域
- 线条生成后不可继续修改

## 6.4 DefenseLineFactory

职责：

- 将玩家绘制点位转换为可碰撞的防御体
- 生成显示节点
- 生成物理碰撞组件
- 处理线条生命周期

实现建议：

- 显示层使用 `Graphics` 绘制
- 碰撞层使用分段矩形 Collider 或链式 Collider
- 线条宽度一期固定，后续可根据道具扩展

## 6.5 ChickenController

职责：

- 控制小鸡移动
- 响应惊吓状态
- 处理碰撞反弹或转向
- 上报死亡事件

移动策略：

- 随机方向移动
- 碰到障碍后改变方向
- 定时微调方向，避免卡死
- 限制最大速度，保证玩家可预判

## 6.6 EagleController

职责：

- 老鹰盘旋
- 选择攻击目标
- 俯冲攻击
- 攻击后回到巡逻点

攻击阶段：

```text
Patrol → LockTarget → Dive → Recover → Patrol
```

一期推荐每只老鹰只攻击一只最近的小鸡，降低复杂度。

## 6.7 CollisionHandler

职责：

- 统一处理碰撞事件
- 判断老鹰是否命中小鸡
- 判断小鸡是否进入危险区域
- 判断防御线是否阻挡老鹰
- 向 `GameManager` 派发胜负事件

# 8. 广告方案

## 8.1 广告类型

一期接入：

- 激励视频广告
- 插屏广告

暂不接入：

- Banner 广告
- 原生模板广告
- 自定义广告位

## 8.2 广告点位

激励视频：

- 失败后复活
- 可选：通关后双倍金币或奖励，若一期没有金币系统则不做

插屏广告：

- 每 3 到 5 关通关后展示一次
- 连续失败时不弹插屏，避免打断重试
- 新用户前 3 关不展示插屏

## 8.3 广告服务接口

```ts
interface AdService {
  init(): void;
  showRewardVideo(scene: string): Promise<boolean>;
  showInterstitial(scene: string): Promise<boolean>;
}
```

返回值说明：

- `true`：广告完整观看或展示成功
- `false`：用户中断、广告未加载、平台不支持

# 9. 平台适配方案

## 9.1 适配层接口

```ts
interface PlatformAdapter {
  getPlatform(): 'wechat' | 'douyin' | 'web';
  getStorage<T>(key: string, defaultValue: T): T;
  setStorage<T>(key: string, value: T): void;
  showRewardVideo(adUnitId: string): Promise<boolean>;
  showInterstitial(adUnitId: string): Promise<boolean>;
  vibrateShort(): void;
  shareAppMessage?(payload: SharePayload): void;
}
```

## 9.2 微信小游戏

需要准备：

- 微信小游戏 AppID
- 激励视频广告位 ID
- 插屏广告位 ID
- 隐私协议配置
- 用户隐私授权弹窗合规处理

## 9.3 抖音小游戏

需要准备：

- 抖音小游戏 AppID
- 激励视频广告位 ID
- 插屏广告位 ID
- 平台隐私与权限说明

## 9.4 构建配置

建议保留两套平台配置：

```text
config/platform.wechat.json
config/platform.douyin.json
```

配置内容：

```json
{
  "platform": "wechat",
  "rewardVideoAdUnitId": "",
  "interstitialAdUnitId": "",
  "enableShare": true,
  "enableVibrate": true
}
```

真实广告位 ID 不提交到公开仓库，可通过本地配置或构建脚本注入。

# 11. 资源与性能方案

## 11.1 美术资源

### 11.1.1 视觉风格边界

一期美术统一采用 Q 版卡通农场风，作为 AI 生成资源、UI 设计和关卡场景的验收基准。

视觉关键词：

- 高饱和
- 圆润
- 可爱
- 农场
- 木头
- 草地
- 夸张表情

世界观：

- 小鸡农场遭到老鹰袭击
- 玩家通过画线生成木栅栏或藤蔓防线
- 防线用于保护惊慌乱跑的小鸡，坚持到倒计时结束

色彩与氛围：

- 主色以天空蓝、草地绿、木头棕、藤蔓绿、暖黄色为主
- 整体明亮、轻松、可爱
- 可以有紧张感，但不能恐怖、血腥或写实

角色与场景方向：

- 小鸡：圆滚滚、黄色、大眼睛、惊慌表情明显，失败表现不做血腥效果
- 老鹰：Q 版但有攻击感，眉毛下压、张嘴、爪子夸张，避免真实猛禽恐怖感
- 防御线：优先使用藤蔓作为主视觉，木桩作为节点装饰，线条要粗且碰撞边界清晰
- 场景：农场、草地、木围栏、谷仓、风车、干草堆、木桶、木箱、石头、水坑等
- UI：木牌、圆角按钮、星星、绿色提示按钮、橙色奖励按钮，按钮尺寸适合横屏小游戏触控

素材生成要求：

- 角色、按钮、图标、道具优先生成透明背景 PNG，便于 Cocos 中拆分使用
- 背景可以使用整图，也可以按远景、地面、装饰物分层
- 一期固定横屏单屏视角，默认不做镜头移动
- 避免写实、暗黑、血腥、硬核战争风格
- 避免在角色、敌人、场景表达上过度接近《Save The Pets》的蜜蜂或宠物包装

### 11.1.2 资源清单

一期建议控制资源规模：

- 小鸡：待机、移动、受惊、失败
- 老鹰：盘旋、俯冲、恢复
- 栅栏或藤蔓：线条贴图
- 场景：农场背景、边界、障碍物
- UI：按钮、星级、弹窗、进度条

## 11.2 音频资源

一期音频：

- 背景音乐 1 首
- 点击音效
- 画线音效
- 老鹰俯冲音效
- 通关音效
- 失败音效

## 11.3 性能控制

- 控制纹理尺寸，优先使用图集
- 单关节点数量保持可控
- 防御线分段数量限制在合理范围
- 老鹰和小鸡数量一期不宜过多
- 关卡切换时主动释放临时节点
- 避免每帧频繁创建对象

# 14. 验收标准

## 14.1 功能验收

- 首页可进入游戏

# 15. 风险与应对

## 15.1 画线碰撞不稳定

风险：

- 线条太细导致老鹰穿透
- 分段 Collider 过多导致性能下降

应对：

- 固定线条宽度下限
- 限制采样点数量
- 高速俯冲时增加碰撞检测频率或使用射线辅助检测

## 15.2 关卡难度不稳定

风险：

- 随机移动导致同一关难度波动过大

应对：

- 关卡内使用可控随机种子
- 限制小鸡初始方向和速度范围
- 前 10 关降低随机性

## 15.3 广告影响留存

风险：

- 插屏过频导致流失

应对：

- 新用户前 3 关不展示插屏
- 插屏间隔至少 3 关
- 失败重试不强弹插屏

## 15.4 双平台 API 差异

风险：

- 微信和抖音广告、存储、分享 API 行为不一致

应对：

- 所有平台能力统一封装到 `PlatformAdapter`
- 游戏业务只依赖统一接口
- 平台不可用时返回失败，不阻断主流程

# 18. 当前结论

一期技术路线建议采用“Cocos Creator 3.8.8 + TypeScript + 本地 JSON 关卡配置 + 平台本地存储 + 广告适配层”的方案。

该方案满足：

- 成本低
- 不依赖后端
- 可快速上线
- 后续可扩展
- 适合先验证小游戏广告收益

当前最关键的技术验证点是画线生成碰撞体、老鹰高速俯冲碰撞稳定性、关卡难度曲线和双平台广告适配。
